# 1.1 Host Header Reflection

## Goal

Identify if Host header is reflected in application responses, indicating trust in the header.

---

## What is Reflection?

Host header reflected when it appears in:
- Absolute URLs in responses
- Redirect Location headers
- Error messages
- HTML content
- JavaScript
- Email links
- Log messages

---

## Testing Process

### Step 1: Baseline Request
```bash
curl -i https://target.com/
```

Note:
- Status code
- Content length
- Location header
- Response content

### Step 2: Test with Different Host
```bash
curl -i -H "Host: attacker.com" https://target.com/
```

### Step 3: Compare

If response differs:
- Different absolute URLs
- Different redirects
- Different content

Then Host header is REFLECTED!

---

## Examples of Reflection

### Example 1: Absolute URL in HTML
```html
<!-- Original with correct Host -->
<a href="https://target.com/profile">Profile</a>

<!-- With Host: attacker.com -->
<a href="https://attacker.com/profile">Profile</a>
```

### Example 2: Location Header Redirect
```
Original request:
Host: target.com
Response: Location: https://target.com/login

Attacker request:
Host: attacker.com
Response: Location: https://attacker.com/login
```

### Example 3: Password Reset Link
```
Original:
To reset password, click: https://target.com/reset?token=abc123

With Host: attacker.com:
To reset password, click: https://attacker.com/reset?token=abc123
```

---

## Where to Look for Reflection

1. **Response Body**
   - Absolute URLs
   - Links
   - Redirects
   - Email content

2. **Headers**
   - Location header
   - Set-Cookie domain
   - Link headers

3. **Error Pages**
   - Error messages
   - Stack traces
   - Debug info

4. **Emails**
   - Password reset links
   - Verification emails
   - Notification emails

---

## Indicators of Reflection

**Vulnerable if:**
- ✅ Absolute URLs use attacker's Host
- ✅ Location header redirects to attacker domain
- ✅ Error messages include Host header value
- ✅ Email links use attacker domain
- ✅ JavaScript uses Host header in URLs

---

## Testing Checklist

- [ ] Send baseline request to target.com
- [ ] Note response content
- [ ] Send request with Host: attacker.com
- [ ] Compare responses
- [ ] Check if Host reflected in URLs
- [ ] Check if Location headers use Host
- [ ] Check if emails use Host
- [ ] Document all reflection points

## Real-World Scenario

```
1. Attacker sends: Host: attacker.com
2. Application returns: 
   "Reset link: https://attacker.com/reset?token=xyz"
3. User receives email with attacker.com link
4. Clicks link thinking it's legitimate
5. Attacker captures token
6. Takes over account!
```
